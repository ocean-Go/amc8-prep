# 家庭 AMC8 备考系统全栈设计方案（含 Google NotebookLM 数据源接入）

> 面向拥有 1-2 名需备考 AMC8 竞赛子女的家庭，聚焦「专业 AMC8 备考能力 + 多子女数据隔离 + 高质量知识来源」。

## 一、项目定位与目标

- **目标用户**：负责监督的家长及其子女（小学至初中）。
- **核心痛点**：多子女学习数据混乱、缺乏系统化备考工具、进度难以追踪、资料来源分散。
- **解决方案**：
  - 登录前：强调价值与功能。
  - 登录后：提供多账户管理、个性化学习路径和家长监督工具。
  - 外部知识：接入 NotebookLM 作为权威学习资料聚合与问答来源。

## 二、整体架构

| 架构层 | 核心职责 | 关键特性 |
|---|---|---|
| 前端层 | 用户交互与视觉展示 | 登录前场景化展示、登录后多账户仪表盘与学习工具 |
| 后端层 | 业务逻辑与权限控制 | 多账户认证与数据隔离、备考业务逻辑、家长权限管控 |
| 数据层 | 数据存储与关联 | 结构化学习数据、非结构化资料、NotebookLM 知识索引元数据 |
| 外部知识层 | 外部内容服务 | Google NotebookLM 知识库接入、来源追踪、可解释引用 |

## 三、前端设计

### 1) 登录前功能展示层

- 家庭场景故事墙、功能交互预览与学习价值数据可视化。
- 清晰登录入口与信任背书，突出与 AMC8 大纲的对齐度与用户反馈。
- 新增「数据源说明卡片」：展示系统内容来自 AMC8 真题、家庭自有笔记和 NotebookLM 管理资料集合。

### 2) 登录后核心功能层

- 多账户体系：家长管理员与子女学习账户，支持一键切换并确保数据隔离。
- 子女学习流程：知识点学习、真题模考、智能错题本、学习报告可视化。
- 家长管理功能：子女对比报表、学习行为跟踪、学习计划与权限设置。
- NotebookLM 学习助手：
  - 子女端可基于当前知识点提问。
  - 回答附带来源摘要与引用标签（例如“来自 2020 AMC8 讲解笔记”）。
  - 家长端可查看提问热词，识别薄弱知识点。

## 四、后端设计

### 1) 认证与权限

- 基于 JWT 的 token 机制，结合角色字段与接口拦截实现数据隔离。
- NotebookLM 相关接口按 `parent_id / child_id` 做访问约束，避免跨家庭数据泄露。

### 2) 备考业务逻辑

- 知识点库三级结构，模考自动判分与错题关联，基于记忆曲线的复习提醒。
- 新增 `knowledge_source_service`：将本地知识点与 NotebookLM 来源映射，支持按模块检索（代数/几何/数论/组合）。

### 3) NotebookLM 数据源接入（关键）

- 接入目标：将 NotebookLM 作为「学习资料索引与问答增强」数据源，而非替代核心业务数据库。
- 接入流程：
  1. 家长在系统中绑定 Google 账号并授权访问 NotebookLM 工作区。
  2. 家长选择 AMC8 相关资料（PDF、讲义、错题整理）同步到家庭知识空间。
  3. 后端定时拉取来源元数据（标题、更新时间、来源类型、标签）并建立本地索引。
  4. 学习端发起问答时，先定位子女当前知识点，再调用 NotebookLM 检索增强并返回带引用答案。
- 输出约束：
  - 默认仅返回与 AMC8 相关来源。
  - 回答必须携带来源引用，避免“无依据回答”。
  - 记录 query 日志用于学习分析，但不保存未脱敏个人隐私文本。

### 4) 数据统计报表

- 实时统计学习数据，生成日/周/月多维报表。
- 新增「知识来源贡献度」指标：统计 NotebookLM 引用内容对答题正确率提升的关联趋势。

## 五、数据层设计

### 1) 核心表

- `users`、`knowledge_points`、`learning_progress`、`exam_records`、`mistake_notebook`、`learning_plans`。
- 新增 NotebookLM 相关表：
  - `notebooklm_connections`：家庭级授权关系、token 状态、同步状态。
  - `notebooklm_sources`：来源文档元数据、标签、可见范围。
  - `qa_source_citations`：问答记录与来源引用映射，支持审计与可解释性。

### 2) 非结构化数据

- 对象存储保存 PDF 与图片，数据库存储访问 URL 并进行权限控制。
- NotebookLM 来源仅保存必要元数据与引用，不复制受限制原文，降低版权与泄露风险。

## 六、技术栈

- 前端：Vue3 + Vite + Element Plus（可按当前工程形态映射到 React/Next.js）。
- 后端：Node.js（Express/NestJS）。
- 数据库：PostgreSQL；对象存储：MinIO；认证：JWT + Redis。
- 外部集成：Google OAuth2 + NotebookLM 接口（通过后端服务统一封装）。

## 七、数据安全

- 通过 `child_id / parent_id` 实现数据隔离，密码使用 BCrypt 加密，传输全程 HTTPS。
- NotebookLM 授权 token 使用服务端加密存储（如 KMS/环境密钥），并设置最小权限与定期轮换。
- 审计日志记录来源访问与引用行为，支持家庭级追溯。

## 八、实施阶段

1. 第一阶段（4 周）：搭建基础框架与登录前展示，完成认证接口与核心表。
2. 第二阶段（6 周）：实现多账户核心功能、备考流程与家长报表。
3. 第三阶段（3 周）：接入 NotebookLM 数据源、上线来源引用问答、完善响应式体验与性能优化。

## 九、关键成功指标

- 覆盖 AMC8 全部知识点与历年真题。
- 数据隔离无越权，关键操作响应时间 < 500ms。
- NotebookLM 问答结果引用率 100%（每条回答至少 1 条来源）。
- 用户使用 1 个月后知识掌握率平均提升 ≥ 15%。

## 十一、架构报告（Architecture Report）

### 1) 分层架构遵循性

建议按以下分层落地，并在代码仓中强制目录边界：

- `presentation`（前端 UI / API controller）：仅处理输入输出与 DTO 映射。
- `application`（use-case/service）：编排业务流程，不依赖具体基础设施实现。
- `domain`（实体/聚合/领域规则）：核心业务规则，不依赖框架。
- `infrastructure`（DB/对象存储/第三方 API）：通过接口适配器实现。

### 2) 无循环依赖（No Circular Dependencies）

建议启用静态依赖扫描（如 `madge` / `dependency-cruiser` / eslint import rules），规则为：

- `domain` 不依赖其他层。
- `application` 可依赖 `domain`，不可反向被 `domain` 依赖。
- `infrastructure` 仅实现 `application/domain` 定义的端口接口。

### 3) 模块松耦合（Loose Coupling）

围绕业务能力拆分模块：

- `identity`（家长/子女账户与权限）
- `learning`（知识点学习进度）
- `exam`（模考与评分）
- `mistakebook`（错题与复习）
- `knowledge-source`（NotebookLM 连接与引用）

模块间通过接口协议与事件协作，避免跨模块直接操作对方数据表。

## 十二、重构建议（Refactoring Suggestions）

1. 先建工程骨架再开发功能：优先初始化 monorepo 或前后端双仓标准目录与 CI。
2. 先定义端口接口：对 NotebookLM、对象存储、题库服务抽象 Provider 接口，再接具体实现。
3. 统一契约层：引入 OpenAPI / JSON Schema 作为前后端契约。
4. 可观测性前置：第一版即落地日志追踪 ID、审计日志与基础指标（QPS/延迟/错误率）。
5. 测试金字塔落地：
   - 单元测试覆盖 domain/application 核心规则。
   - 集成测试覆盖数据库与权限隔离。
   - E2E 覆盖主链路（家长创建子女账户→学习→模考→错题复习）。
